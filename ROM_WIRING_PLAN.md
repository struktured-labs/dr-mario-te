# ROM wiring plan (ultracode wf_b1dd28df, 2026-06-28)

Verdicts: all 3 FLAWED-but-architecture-SOUND. Refinements: sub-pass cascade yielding, dead-RAM relocation of cross-frame state, clear $F6 in HOLD, emit w_act unconditionally.

## execution_model

RUN THE SLICE IN THE NMI CONTROLLER-HOOK (option iv), never the $B660 main-loop spin. The wrapper at $FF54 (patch_vs_cpu.py _emit_rotation_wrapper, RELOC_CPU=0xFF54) lives in the FIXED bank ($C000-$FFFF, always mapped) and is already reached in NMI context (fact 6). It calls the unit1 slicer through two tiny FIXED-BANK helpers `sel_unit1`/`sel_unit0` (RTS-returning clones of the validated $D2CC 5-bit serial sequence; do NOT reuse $D2CC itself — it ends in JMP $FB00, wrong return semantics for a mid-wrapper subroutine). Per hook: [read P2 colors to RAM]; on new-pill edge `JSR sel_unit1 / JSR arm / JSR sel_unit0`; every frame `JSR sel_unit1 / JSR slice / JSR sel_unit0`; then the ST_MODE gate. WHY IT CANNOT CRASH FROM THE ASYNC-NMI-VS-BANK HAZARD: unit1 is mapped ONLY in the span between the two FIXED-BANK serial writes, which lies entirely inside ONE NMI hook; we bound total per-hook work to < 1 frame (hang_proof), and fact 4 guarantees the next NMI is ~29780 cyc away, so unit0 is always restored (and the hook RTId) before any next NMI can fetch $8005 — there is no instant at which $8000=unit1 while an NMI could fire. The bank-switch opcodes are fetched from the fixed bank, so the executing half is never the half being switched. REJECT the spin (options ii/iii): an async NMI landing mid 5-write serial sequence corrupts the MMC1 shift register, and after a unit1-restoring $8005 stub the game's RTI returns into the interrupted unit1 PC now remapped to unit0 garbage = crash; masking NMI for a whole ~114k-173k cascade loses $33/audio for 4-6 frames. BUDGET RE-FRAMING that makes the NMI free: INTEGRATION_SPEC's spin choice conflated the 2273-cyc figure (the vblank PPU-WRITE window, §7) with the COMPUTE budget. The search does ZERO PPU writes; the controller hook computes with the full inter-NMI frame (~29780 cyc) as its only deadline — exactly why v28cs already runs its whole depth-1 search in this same hook and plays L11 identically (fact 4). BONUS: arm and slice now run single-threaded in the same NMI, so the prior arm-vs-slice cross-context race and the $DD/$DA torn read both vanish (no preemption point between them); ST_BUSY ($0199) is kept only as defense-in-depth.

## cascade_handling

RESUMABLE CASCADE at pass granularity is PRIMARY; a pass-cap is only a liveness backstop. The validated slicer does one ATOMIC eval per slice (test_slicer.py line 189) and that eval includes the full resolve_targeted cascade (primitives.py kernel line 263 -> resolve_targeted rt_loop lines 67-75), which measures up to ~114k-173k cyc (DEPTH2 §table: resolve 32k avg/114k max) = 4-6 frames. Atomic-in-NMI therefore OVERRUNS -> nested-NMI hang. The validation only checked correctness + one-eval-per-slice, NOT cycle budget, so this is THE gap. FIX: restructure resolve_targeted so the slicer drives the cascade ONE find_clears+gravity pass per slice. Add sub-state in the reserved block: ST_SUB $019A (0=fresh/1=cascading), ST_PASS $019B. Flow: FRESH (ST_SUB=0) -> advance to next legal cursor, place 2 cells in a PRIVATE BOARD buffer, run the cheap targeted first pass (find_clears_targeted, lines 130+, only the 2 placed lines, ~3-8k); NO-CLEAR (~95%) -> shape+score, advance cursor, yield (one cheap eval, unchanged behavior); CLEAR -> gravity, ST_SUB=1, ST_PASS=1, yield WITHOUT advancing. CASCADING (ST_SUB=1) -> one full-board find_clears + gravity per slice (the rt_loop body), INC ST_PASS; still clearing -> yield; settled -> shape+score, ST_SUB=0, advance cursor, yield. Because the SAME passes run in the SAME order, just time-sliced across NMIs, the final board+score is BIT-IDENTICAL to monolithic: re-run test_resolve (500/500) and test_slicer (400/400) to confirm the restructure is transparent. CRITICAL: BOARD must be a slicer-PRIVATE 128B buffer (+128B SCRATCH), NOT the live rendered playfield — the kernel leaves BOARD mid-mutated across a mid-cascade yield (the restore at kernel lines 265-267 hasn't run), so if BOARD were live RAM the game would render a corrupt board for those frames. arm snapshots the gravity-frozen settled playfield into BOARD ONCE; SCRATCH/RV_CELLS/RV_VIR (zp) persist across yields naturally. MEASURE single-pass cyc on py65: if the worst full-board pass (find_clears+gravity) exceeds the ~18k safe quantum, ESCALATE to row/col-granular yield inside find_clears (the loop is already _ROW/_COL-cursored at primitives.py 81-92; save MARK $0180-$018F + the 12B fc pool to the $01A0-$01AB guard band, resume next NMI). CAP = (golden max pass count over the 400 boards)+margin, force-scoring the partial board on a cap-hit (graceful, no infinite loop) — set high enough that it NEVER fires on the validated set, so it adds zero golden divergence and exists only to terminate pathological live boards. REJECT cap-as-primary (design 1): capping mid-cascade changes argmax on the ~5% clearing boards and can drop a winning clear, degrading the validated 71% clear / 17% win — unacceptable when resumable preserves it exactly.

## bank_hygiene

Bank-restore is STRUCTURAL: every unit1 JSR in the wrapper is bracketed `JSR sel_unit1 / JSR <unit1 entry> / JSR sel_unit0`, with sel_unit0 in the FIXED bank executing immediately after the unit1 routine RTSes back. So EVERY slicer exit path — ST_BUSY bail (test_slicer 58), ST_MODE!=SEARCHING bail (59), sl_yield (67-68), publish+DONE (147) — returns into the fixed-bank wrapper which unconditionally restores unit0 before its own RTS; no path can leak unit1. arm and slice each get exactly one sel_unit1/sel_unit0 pair. TRAMPOLINE/STUBS LOCATION: sel_unit1, sel_unit0, the $FF54 wrapper, and the ST_MODE gate live in the FIXED bank ($C000-$FFFF, the ~392B fragmented free space + the dead-v17 window at $FF54/0x7F64, assert-checked at patch_vs_cpu.py 1148); the slicer (arm, slice, try_eval, eval_one, advance, publish, kernel, resolve_targeted, find_clears(_targeted), gravity, shape, first_occ + the BITS/MARK tables + the private 128B BOARD and 128B SCRATCH buffers) lives in PRG unit1 $8000-$BFFF. SERIAL-WRITE SEQUENCE (clone the validated $D2CC bytes verbatim, just wrap as RTS-returning helpers): reset the MMC1 shift register with one write that has bit7=1 to $8000-$FFFF, then 5 writes LSB-first with bit7=0 to a $E000-region address in the always-mapped fixed half — value 0 (five $00 writes) selects unit0; value 2 (bit0 sequence 0,1,0,0,0) selects unit1. All 5 writes execute inside the NMI hook, so no nested NMI (fact 4) can land mid-sequence and latch a corrupt value — the exact failure that kills the main-loop-spin variant. Add an assembler assert that no slicer store targets $01A0-$01EE except the documented 12B guard-band cascade-save area. Confirm the assembled fixed-bank additions fit before flashing.

## hang_proof

Both fatal modes reduce to ONE bound: per-hook work < inter-NMI interval (~29780 NTSC cyc). (1) NO NESTED-NMI HANG: per hook = game NMI/render baseline (X, MEASURE on Mesen) + wrapper (~0.3k) + sel_unit1+sel_unit0 (~0.5k) + at most one arm (~0.1k, edge frames only) + ONE slice. By cascade_handling a slice is at most ONE pass: NO-CLEAR targeted ~3-8k, or one cascading full-board find_clears+gravity bounded to the ~18k safe quantum (with row-granular yield as escalation if a pass measures higher). Worst slice <= ~18k -> total <= X + ~19k. v28cs already runs its FULL search in this same hook with no mode-255, which is the empirical proof X+search fits one frame; our per-hook work is a bounded subset. ASSERT max hook span < ~28k on Mesen (=> X < ~9k headroom, met). Because each hook RTIs before the next NMI, NMIs never nest, the stack never compounds, mode-255 is impossible. The ~114k-173k cascade can NEVER land in one NMI: it is spread one pass/frame across ~4-9 frames and the cap guarantees termination. (2) NO VECTOR-INTO-UNIT1 CRASH: unit1 is mapped only between the two fixed-bank serial writes inside one hook; the same <1-frame bound guarantees unit0 is restored and the hook RTIs before any next NMI fetches $8005. The main loop NEVER runs with unit1 mapped, so every async main-loop NMI vectors into unit0's valid $8005; the bank-switch opcodes are fetched from the fixed bank, so the executing half is never unmapped mid-switch. (3) STACK (no nesting -> no accumulation): deepest chain wrapper->sel(returned)->slice->kernel->resolve_targeted->find_clears->fc_scan->fc_apply = ~7 JSR = 14B of return addrs, zero PHA/PHP in the sim primitives, atop the game NMI baseline (~17-18B incl. trampoline PHA) -> worst SP ~$01DC, ~60B above the $019F live-RAM ceiling (+ up to $01AB if the 12B guard-band save is used). ASSERT min SP > $01C0 on Mesen across a deep-cascade NMI AND a level-clear multi-SFX audio frame.

## Verdicts (all FLAWED -> refinements, arch sound)

### 0: flawed

The plan's two architectural choices are SOUND, but its hang/crash SAFETY PROOF is not provable as written — and the project's own measured data breaks the committed cascade granularity.

WHAT IS SOUND (do not relitigate):
- execution_model option (iv): run the slice in the NMI controller-hook, never the $B660 spin. Correct. Rejecting the spin is right — an async NMI mid 5-write MMC1 serial sequence corrupts the shift register, and a unit1-restoring $8005 stub still RTIs into a remapped PC. The fixed-bank sel_unit1/sel_unit0 bracket (RTS clones of $D2CC, not $D2CC itself) is the right structure.
- The reduction "both fatal modes = ONE bound (per-hook work < inter-NMI interval)" is CORRECT. Because the NMI/$FFFA path vectors to $8005 in the switchable half (fact 3), the bank-crash window is exactly the span $8000=unit1, which lies entirely between the two fixed-bank serial writes inside one hook. If the hook finishes before the next NMI, neither a nested NMI (hang) nor a vector-into-unit1 (crash) can occur. So the entire safety case collapses to: every slice's worst-case work < the inter-NMI interval.
- The NMI budget REFRAMING is also legitimate: a handler doing zero PPU writes is bound by the full inter-NMI interval (~29830 cyc, DEPTH2 line 147), not the 2273-cyc vblank PPU window. v28cs running depth-1 in this hook supports it (caveat below).

WHY IT IS NOT PROVABLY SAFE (the decisive adversarial finding):
The plan's committed cascade granularity is "one full-board find_clears + gravity pass per slice," asserted to fit a "~18k safe quantum." The project's OWN measured table (DEPTH2_FEASIBILITY.md) contradicts this:
- find_clears full-board = ~25k cyc (line 71)
- apply_gravity = ~5.9k cyc (line 72)
- => ONE cascading pass = find_clears + gravity ≈ ~31k cyc
- inter-NMI interval = ~29830 cyc (line 147)
- resolve = ~32k avg / 114k max (line 73), i.e. multiple such passes.

A single cascading pass (~31k) ALONE exceeds the entire ~29830-cyc inter-NMI interval — before adding ANY baseline NMI render work X. find_clears is structurally fixed-cost (24 fc_scan calls over fixed rows/cols at primitives.py 78-93 + fc_apply walking all 128 cells at 105-125); it does not get cheaper as the board empties mid-cascade. So on the ~5% clearing placements (and the cluttered L11 endgame is exactly where clears happen), the committed slice OVERRUNS the frame. That is precisely the dual-fatal interleaving the skeptic was asked to find: the next NMI fires while the hook is still inside the sel_unit1/sel_unit0 bracket with $8000=unit1 -> simultaneously a nested-NMI stack compounding (mode 255 hang) AND a fetch of the NMI handler at $8005 from unit1 garbage (crash). Worst case: a full-board pass on the first clear of a dense board, with the NMI landing ~29830 cyc in, mid find_clears.

The plan ACKNOWLEDGES this only obliquely: it offers "row/col-granular yield inside find_clears" as an ESCALATION "if the worst full-board pass exceeds the ~18k safe quantum." The measured 25k find_clears means that escalation is not a contingency — it is MANDATORY on every cascading slice — yet it is left unspecified (which scan state: MARK $0180-$018F, _RUN/_MCOL/_RSTART/_OFF/_CNT accumulators, the fc_scan cursor, the fc_apply 0..127 walk, plus per-column gravity cursor _GCOL/_GREAD/_GDEST) and unvalidated. The harder problem (resuming a half-finished find_clears scan and a half-finished gravity compaction across NMI boundaries, bit-identically) is the actual make-or-break, and the plan treats it as a footnote.

1. The artifact called "the VALIDATED slicer (400/400)" is NOT resumable at pass granularity. test_slicer.py + primitives.py resolve_targeted (rt_loop, lines 67-75) run the FULL cascade ATOMICALLY inside one eval_one inside one slice; the kernel (lines 257-268) backs up BOARD->SCRATCH and restores at the end as one indivisible unit. Also: w_act is placed INSIDE the `if rotate_exec:` block, so the gate is only well-formed for rotate_exec=True (the ship config); with rotation disabled, BEQ w_act references a missing label and the move/drop runs ungated.

3) ORIENT ENCODING / COLOR ORDER: this is the headline mis-colored-move risk, and the plan over-cites its evidence. But the claim that the 400/400 run "asserts" {0,3} is FALSE: line 195 `exp_da = 0 if SE_BORIENT==1 else 3` then line 196 just checks PUB_DA==exp_da — a tautology re-deriving publish's own logic. It proves NOTHING about the physical $03A5 color order. CONCRETE MIS-COLORED CASE: search always places PCA at offa = UPPER for vertical (te_eval offa=offb-8, lines 79-81). With wrapper edit (a) SE_PCA<-$0381, SE_PCB<-$0382. For a vertical pick with pca!=pcb (~2/3 of pills), the actuator rotates CCW 0->3 and the search PREDICTED PCA($0381) on top. ### 2: flawed

DETERMINISM LENS: The architecture is decision-transparent IN PRINCIPLE — because every byte of slicer state is in RAM and no branch reads a frame-counter/wall-clock, time-slicing arm->slice* across NMIs is bit-identical to the py65 driver, which itself already does call(arm) then a synchronous call(slice) loop (test_slicer.py 186-189). The pass-order claim also holds: FRESH=find_clears_targeted, CASCADING=full find_clears mirrors resolve_targeted (primitives.py 60 vs 68). So the plan CAN reproduce 400/400 — but as SPECIFIED it does not robustly guarantee it. Four divergence/corruption holes:

1) LOAD-BEARING GAP — cross-frame clobber of the persistent working set (the real "board-change-mid-search"). The validated kernel is ATOMIC (primitives.py 257-268: backup BOARD->SCRATCH ... resolve ... restore), so BOARD($0500), SCRATCH($0600), MARK($0300) and the zp cascade temps ($D9-$E1: Z_TILEA/B, Z_OFFA/B, RV_CELLS/VIR, PASS_*, _ROW.._FLCNT) are touched only within ONE NMI; the game's main loop and its own render NMI can never interleave. The resumable restructure PROMOTES these to must-persist-across-frames, but they were validated only in py65 isolation where nothing runs between calls. On the cartridge the game's main loop + render run between every slice, and Dr. Mario uses zero page heavily; any game write to $D9-$E1 / $0300 / $0500-$067F mid-cascade corrupts the carried board/totals -> silent argmax divergence OR a malformed cell -> longer cascade -> cap-hit or crash. The plan's "SCRATCH/RV_CELLS/RV_VIR persist across yields naturally" is FALSE on hardware — "naturally" only held under atomicity. This is THE determinism hole the schedule restructure introduces, and it is unaddressed (the plan only reserved the DRIVER state $0190-$019B, not the cascade working set).

2) CAP MIS-CALIBRATION admits live-board divergence. The plan sets cap = (golden max passes over the 400 random boards)+margin, which guarantees zero divergence only on the validated set. Live L11 VS-CPU boards (incl. incoming garbage) are NOT drawn from rand_board(rng)/BIAS and can exceed the empirical max -> cap fires mid-cascade -> the ~5% clearing branch can drop a winning clear, exactly the failure the plan claims to avoid. The cap should be the COMBINATORIAL maximum: each find_clears pass clears >=4 of <=128 cells (else PASS_CELLS=0 and rt_loop terminates), so <=32 passes bounds ANY reachable board. Cap ~40 -> can never fire on any board -> pure liveness backstop with provably zero golden divergence on hardware.

3) VALIDATION HARNESS will spuriously FAIL the restructure. test_slicer.py bails at nsl<80 (line 188); pass-granular slicing multiplies slice count (15 cursors + clearing cursors costing up to ~32 slices each -> hundreds), so the loop hits nsl=80 before ST_MODE=DONE and reads non-final SE_BCOL/SE_BORIENT -> mass MISMATCH for a NON-determinism reason. "Re-run test_slicer 400/400 to confirm transparency" is invalid until the nsl bound is raised and the one-eval-per-slice invariant is replaced with one-pass-per-slice. golden_search equality stays valid; only the scaffold must change.

4) CASCADING-slice gravity must stay CONDITIONAL. rt_loop runs gravity only when find_clears found cells (primitives.py 68-74); the plan prose "one full-board find_clears + gravity per slice" reads as unconditional. An unconditional extra gravity on the settling pass is idempotent in THIS code (settled board has nothing to fall) so benign, but it is a latent divergence and the implementer must replicate the find_clears-result-gates-gravity ordering exactly.

Minor: bank_hygiene says BOARD/SCRATCH "live in PRG unit1 $8000-$BFFF" — category error; they are writable RAM ($0500/$0600), cannot reside in switchable PRG ROM (only the CODE is in unit1). The escalation cites "MARK $0180-$018F" but MARK is $0300; the kernel docstring's "$6D/$6E/$D2/$D3" is stale vs real $DC/$DE/$D9/$DB. The orient encoding {vert->$DA=3, horiz->0} DOES match publish (test_slicer.py 162-167) and the harness exp_da assertion (line 195), so that piece is sound and needs no publish edit.

**Fix:** To make scheduled/capped execution provably reproduce 400/400 on hardware:

A) RELOCATE + AUDIT the cross-frame persistent set. Before flashing, audit Dr. Mario's RAM map and move every byte the resumable cascade carries across NMIs into verified-DEAD RAM: 128B BOARD, 128B SCRATCH, 16B MARK, and the zp carry-state that actually lives at pass boundaries (RV_CELLS/RV_VIR $E0/$E1, Z_OFFA/B $DC/$DE, Z_TILEA/B $D9/$DB) plus driver $0190-$019B. Confirm via a Mesen write-breakpoint watchlist that NO game code writes any of these for the full multi-frame span of a held search (same run that proves gravity-freeze). Keep PASS granularity (avoid splitting find_clears across NMIs) precisely so per-pass temps (MARK, _ROW.._FLCNT) never need to persist; a full-board find_clears+gravity over 128 cells is a few thousand cycles, well under one frame, so the row/col escalation is unnecessary and should be dropped.

B) CALIBRATE THE CAP from the combinatorial bound, not the empirical 400-max: cap >= 32 passes (each pass removes >=4 of <=128 cells). Set ~40. Prove on the py65 histogram that the worst of the 400 boards is far below the cap (it will be), so the cap adds zero golden divergence on ANY board, not just the validated set. Keep the graceful force-score on cap-hit only as an anti-hang terminator.

C) UPDATE THE HARNESS before claiming transparency: raise test_slicer's nsl bound to >= (15 + 12*cap); change the resumability assertion from one-eval-per-slice to one-pass-per-slice; keep golden_search/exp_da equality unchanged. Add a test_resolve variant that drives the cascade one pass per call and asserts bit-identical final board+totals vs monolithic resolve_targeted (the real transparency proof). Only after both pass does 400/400 + 500/500 certify the restructure.

D) ENFORCE conditional gravity in CASCADING: replicate rt_loop exactly — find_clears; if PASS_CELLS==0 -> settle (shape+score, advance, ST_SUB=0); else accumulate totals, gravity, INC ST_PASS, yield.

E) Fix the BOARD/SCRATCH-in-ROM wording to RAM, and verify on Mesen the arm-time snapshot is taken on a fully settled post-cascade playfield (new-pill P2y edge) so find_clears_targeted's "no pre-existing >=4 run" precondition (primitives.py 129) holds — the same assumption golden_search makes, so no new divergence once confirmed.

With A-E the time-slice is decision-transparent and the cap provably never fires on any reachable board, so scheduled/capped execution reproduces the validated argmax exactly. Without A and B specifically, the design admits both silent divergence and crashes on live boards — so the canonical plan's determinism guarantee is not yet sound as written.
